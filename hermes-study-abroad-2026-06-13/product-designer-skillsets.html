Product Designer 공고 요구 스킬셋 정리 — 2026-06-14

성격: 공개 영문 공고/직무 설명 기반의 의사결정·커리어 준비용 요약. 개인 식별정보나 회사 내부자료 미포함.

핵심 결론

미국/글로벌 Product Designer 공고에서 요구하는 역량은 단순 UI 제작보다 제품 문제 정의 → 리서치/데이터 → 플로우/프로토타입 → 시스템화 → 엔지니어 협업 → 임팩트 커뮤니케이션으로 수렴한다.

특히 사용자 현재 맥락에서는 “일반 Product Designer”보다 아래 포지셔닝이 더 맞다.

AI-aware Senior Product Designer / Design Technologist / UX Engineer-adjacent Product Designer

이유: - 삼성 경력은 junior가 아니라 senior/complex-product 포지션의 원재료다. - SVA IxD + 국비 $100K는 “미국식 공개 포트폴리오와 네트워크”를 만드는 장치로 써야 ROI가 난다. - H-1B/취업 관점에서는 저임금 generalist보다 고임금·전문화·시니어 포지션이 유리하다.

1. 공고에서 반복되는 8대 스킬셋

스킬셋 공고 표현 포트폴리오에서 보여줄 증거
Product thinking business goals, product strategy, measurable impact, roadmap 문제를 왜 풀었는지, 성공 지표가 무엇인지, 결과가 무엇이었는지
UX / interaction design user flows, IA, wireframes, interaction flows, intuitive experience 복잡한 문제를 간단한 flow로 바꾼 before/after
Visual/UI craft high-fidelity mockups, typography, layout, visual systems 완성도 높은 화면, 컴포넌트, 상태값, 반응형 처리
Research & validation user research, usability testing, customer feedback, market analysis 인터뷰/테스트가 실제 디자인 결정을 바꾼 사례
Prototyping interactive prototypes, click-throughs, advanced prototyping Figma prototype, coded prototype, AI/LLM prototype
Design systems scalable components, patterns, UI specs, consistency 컴포넌트 구조, 토큰, 사용 규칙, 엔지니어 handoff
Cross-functional collaboration PM/engineering/research collaboration, stakeholder communication PM/개발자와 trade-off 조정한 의사결정 기록
Communication / rationale present design rationale, advocate for design, write clearly 선택 이유, 반대안, 측정 근거를 명확히 설명하는 case study

2. 레벨별 요구 차이

Early / junior-leaning UX Designer

Google/YouTube early UX 공고와 Amazon UX internship류에서 반복되는 기준:

  • 포트폴리오 링크 필수
  • 1년 안팎 interaction/product/UX 경험 또는 학업 기반 실무 경험
  • Figma로 user flow, high-fidelity mockup, interactive prototype 제작
  • 사용자 중심 디자인 방법론 이해
  • PM/Engineering과 협업
  • ambiguity를 견디고 피드백을 받아 iteration

사용자에게의 함의: - 이 레벨로 들어가면 “미국 첫 직장”은 가능할 수 있지만, 삼성 경력을 충분히 활용하지 못하고 wage/visa 측면에서 손해일 수 있다.

Mid/Senior Product Designer

Digital Waffle, Meta 계열 공고 요약, KOHO 등에서 반복되는 기준:

  • 3~6년 이상 product/digital design 경험
  • end-to-end feature/product ownership
  • web/mobile product portfolio
  • user research와 usability testing 경험
  • Figma 등 modern design tool 숙련
  • design systems, accessibility, components 이해
  • business goals와 user needs 균형
  • agile/cross-functional 환경에서 PM/engineer/stakeholder 협업
  • 디자인 rationale을 다양한 audience에게 설명

사용자에게의 함의: - 삼성 경력은 이 구간을 노려야 한다. - 단, 삼성 경험이 NDA 때문에 보이지 않으면 senior signal이 약해진다. 그래서 공개 가능한 case 1개가 G5 핵심이다.

Specialized / high-complexity Product Designer

Harmattan AI류 복잡계 공고와 Design Technologist류 공고에서 반복되는 기준:

  • 7~10년 또는 그에 준하는 복잡한 시스템 UX 경험
  • data-heavy / mission-critical / healthcare / defense / trading / enterprise SaaS 등 복잡 도메인
  • workflow simplification
  • design systems + accessibility
  • engineering과 fidelity/feasibility 검증
  • geospatial, dashboard, real-time telemetry, AI-powered product 등 기술 맥락 이해
  • portfolio가 “복잡성을 단순화한 증거”를 보여야 함

사용자에게의 함의: - “삼성 내부 복잡한 시스템/제품 경험 + AI/개발 역량 + IxD”를 잘 번역하면 일반 PD보다 이쪽이 더 차별적이다.

3. AI-era Product Designer에서 추가로 보이는 요구

최근 공고/직무 설명에서 새로 강해진 항목:

요구 의미 사용자 준비 과제
GenAI workflow AI 도구로 ideation, synthesis, prototype 속도 향상 Claude/Figma/프로토타입 workflow를 case study에 명시
AI product understanding AI-powered product의 불확실성, trust, transparency 설계 AI interaction 원칙 정리 + 작은 prototype
Technical literacy LLM/API, frontend, data-heavy UI 이해 React/TS 또는 coded prototype 증거
Human judgment AI 산출물 검증, 큐레이션, 의사결정 “AI를 썼다”가 아니라 “무엇을 검증했고 무엇을 버렸는가” 기록

핵심: AI는 “툴 사용”만으로는 약하다. 공고가 원하는 것은 AI로 더 빨리 그리는 사람보다 AI가 들어간 제품을 믿을 수 있게 설계하는 사람에 가깝다.

3-1. 2026-06-14 공개 JD 재확인 메모

이번 점검에서 Figma의 Product Designer, Design, Dev, & AI Tools와 Epoch AI의 Product Designer 공개 공고를 추가로 확인했다. 반복 패턴은 아래와 같다.

  • Figma: technically-minded product designer, AI/Interactivity/Design Systems/Dev Tools 영역, product strategy, storytelling, PM/Engineering/Research 협업, user flows/prototypes/high-fidelity visuals, discovery-to-launch ownership, user needs·technical constraints·business goals 균형, AI/emerging tools를 활용한 research/prototyping/building 경험, HTML/CSS·advanced Figma·design/development hybrid는 plus.
  • Epoch AI: AI research를 intuitive/high-impact UI·dashboard·data visualization으로 번역, responsive UI/graphs, design specs, mockups/prototypes, design-system consistency, user research/usability testing 가능성, data fluency, Figma auto-layout/components, front-end와 Python visualization tool은 plus.

사용자에게의 직접 함의:

  1. SVA 포트폴리오 1번 케이스는 “예쁜 화면”보다 복잡한 정보/시스템을 단순한 workflow와 의사결정 UI로 번역한 증거가 강하다.
  2. AI-era 포지셔닝은 “AI 툴을 씀”이 아니라 AI/데이터/복잡 시스템을 다루는 제품에서 사용자 신뢰와 판단을 설계함으로 써야 한다.
  3. Figma advanced features + coded/data prototype + clear rationale writing 조합은 일반 PD보다 Design Technologist / UX Engineer-adjacent 포지션 신호를 만든다.

3-2. 2026-06-14 12:15 KST 공개 JD 추가 점검

이번 루프에서는 개인/회사 내부정보 없이 일반 공개 JD만 확인했다. Ashby/Lever/Greenhouse 공개 공고에서 새로 반복 확인된 패턴은 아래와 같다.

  • AI-native workflow designer: 0G Labs·Raspberry AI·Eloquent AI류 공고는 “AI-powered / AI-native product”를 단순 키워드가 아니라 복잡한 기술을 인간이 이해 가능한 workflow, dashboard, CUI/conversational interaction, agent monitoring/control UI로 바꾸는 역량으로 요구한다.
  • Power-user / B2B complexity: EvenUp·Raspberry AI·Finix·JetBrains류 공고는 specialized power users, enterprise SaaS, complex technical domain, developer/user expert workflow를 반복적으로 요구한다. 즉 “대중용 앱 화면”보다 전문가의 반복 업무를 줄이는 정보 구조와 상태 설계가 강한 신호다.
  • Design systems as shipped infrastructure: Finix·Raspberry AI·Figma류 공고는 component libraries, variants/states, token-driven systems, implementation QA, production collaboration을 요구한다. 포트폴리오에서는 “컴포넌트를 만들었다”보다 실제 workflow에 적용되어 일관성/속도/품질을 개선한 증거가 필요하다.
  • Research + writing + rationale: EvenUp·JetBrains·Figma 공고는 user research, usability testing, structured rationale, strong writing을 함께 요구한다. AI 시대에도 최종 차별점은 “무엇을 왜 선택했는지 설명하고 검증한 기록”이다.
  • Technical literacy is now a portfolio signal: HTML/CSS/JS, React, front-end constraints, motion/prototyping tools, data visualization, Python visualization 등은 필수/우대 조건으로 반복된다. 사용자는 coded/data prototype을 1개만 공개해도 일반 PD와 다른 신호를 만들 수 있다.

사용자에게의 업데이트된 결론:

SVA 2년의 산출물은 “미국식 Product Designer 포트폴리오”가 아니라 AI·데이터·복잡 시스템을 사용자 신뢰와 실행 가능한 workflow로 번역한 포트폴리오여야 한다.

G4 alumni 인터뷰에는 아래 질문을 추가한다.

  1. “최근 합격/면접 공고에서 AI product, design systems, coded prototype 중 무엇이 실제로 평가되었나?”
  2. “포트폴리오에서 complex workflow / data-heavy UI / B2B power-user 사례가 general app redesign보다 더 강하게 읽혔나?”
  3. “디자인 rationale writing과 PM/Engineering 협업 사례를 어떻게 증명했나?”

3-3. 2026-06-14 12:31 KST 공개 JD 추가 확인

이번 루프에서는 Figma Product Designer, AI Models, Niural Senior Product Designer, Goodie AI Product Designer 공개 원문을 추가 확인했다. 개인/회사 내부정보는 검색·반영하지 않았다.

  • AI evaluation / heuristics: Figma AI Models 공고는 디자이너가 AI 연구와 디자인 원칙 사이의 bridge가 되어, AI-generated output의 품질을 평가하고 design principles/heuristics를 문서화하는 역할을 요구한다. AI 역량은 “툴 사용”보다 모델 산출물을 판단·평가·가이드하는 디자인 기준으로 옮겨가고 있다.
  • Senior prototype-as-business artifact: Niural은 sales demo, internal showcase, stakeholder presentation에 쓰이는 high-fidelity interactive prototype ownership을 강조한다. 즉 prototype은 학습용 산출물이 아니라 비즈니스 의사결정과 alignment를 움직이는 증거물이어야 한다.
  • B2B AI control plane / dashboard: Goodie AI는 AI search/brand visibility라는 새 B2B 문제공간에서 research, user flows, design systems, data feedback, dashboard/data visualization, accessibility, front-end constraints를 함께 요구한다. 이는 사용자 포트폴리오가 AI/데이터/복잡 workflow를 readable dashboard와 control surface로 번역하는 사례를 가져야 함을 강화한다.

포트폴리오 체크리스트에 추가할 질문:

  1. 내 case에는 AI/데이터 산출물을 평가하는 heuristic 또는 quality bar가 있는가?
  2. prototype이 단순 화면 시연을 넘어 stakeholder decision, PM/Engineering alignment, business metric에 연결되는가?
  3. dashboard/control-plane류 복잡 UI에서 상태, 권한, 예외, 신뢰/검증 흐름을 보여주는가?

3-4. 2026-06-14 12:46 KST 공개 JD 추가 점검

이번 루프에서는 Greenhouse Senior Product Designer, Data, Preql AI Product Designer, Lodgify/Jobgether Senior Product Designer — Design Systems 공개 공고/스니펫을 추가 확인했다.

  • Data product design is a bridge role: Greenhouse·Preql AI류 공고는 designer가 데이터 엔지니어링/스키마/쿼리 가능성, reporting/analytics, 데이터 정제 workflow를 이해하고 PM·Engineering·Data Science와 함께 의사결정을 만들어야 함을 반복한다.
  • Trustworthy AI/data workflow: Preql AI는 finance team이 자연어로 데이터 파이프라인을 만들고, transformation logic을 감사 가능한 방식으로 이해해야 한다고 명시한다. AI UX의 핵심은 “마법 같은 생성”보다 사용자가 결과·근거·수정 과정을 신뢰할 수 있는 설명 구조다.
  • Design system governance: Lodgify/Jobgether류 design-system 공고는 component 제작만이 아니라 contribution model, documentation, governance, adoption, engineering implementation alignment를 요구한다. 포트폴리오에는 “시스템을 만들었다”보다 어떻게 채택·유지·확장되게 했는가가 보여야 한다.

사용자에게의 추가 함의:

  1. 공개 case 1개는 complex data/AI workflow → readable state model → validated prototype → design-system pattern 흐름으로 구성하면 JD 반복 신호와 잘 맞는다.
  2. AI/technical literacy는 개발 지식 과시가 아니라, 기술 제약을 이해해 사용자가 믿고 조작할 수 있는 인터페이스로 번역한 증거로 써야 한다.
  3. G4 alumni 인터뷰에는 “design system이 실제 채용 평가에서 UI craft인지, engineering adoption/governance 증거인지”를 추가로 물어본다.

3-5. 2026-06-14 13:02 KST 공개 JD 추가 점검

이번 루프에서는 개인/회사 내부정보 없이 공개 JD/공개 리소스만 확인했다. 새로 확인한 반복 신호는 아래와 같다.

  • Design system은 이제 UI kit가 아니라 제품 인프라: Tabs와 Onebrief의 design-system 공고는 components·patterns·tokens·documentation뿐 아니라 Figma와 code sync, Storybook/modern frontend, adoption, contribution model, QA, accessibility를 요구한다. 즉 포트폴리오에서는 “컴포넌트 목록”보다 디자인-엔지니어링 drift를 줄이고 실제 ship 속도/품질을 올린 운영 증거가 강하다.
  • AI-native interface는 control/override까지 포함: Samsung Research America의 Staff AI Product Designer 공고는 agentic OS, multi-agent task orchestration, proactive systems, user control/override, multimodal interaction을 명시한다. AI 제품 UX는 화면 생성이 아니라 모델 행동을 사용자가 이해·개입·복구할 수 있게 만드는 상호작용 구조가 핵심이다.
  • AI workflow는 생산성 도구가 아니라 판단 재배치: Figma의 AI for product design 공개 리소스는 research/data synthesis, ideation, prototype acceleration을 강조하지만, JD들과 함께 보면 차별점은 “AI로 빨리 만들었다”가 아니라 AI가 만든 결과를 어떤 기준으로 검증하고 제품 판단에 연결했는가다.
  • 접근성/환경 제약이 senior signal: Onebrief류 공고는 low-vision, small/legacy screens, low-luminance environments, WCAG 2.2, real-world hardware constraints까지 요구한다. 복잡 시스템 포트폴리오에는 edge case·환경 제약·실패 상태 설계가 들어가야 senior하게 읽힌다.

사용자에게의 추가 함의:

  1. 공개 case에는 AI/복잡 시스템 → 사용자 control/override → 예외·복구 흐름 → design-system/token pattern → engineering adoption의 연결을 넣는다.
  2. SVA에서 만들 산출물은 “학교 과제”가 아니라, JD 언어로 번역 가능한 shipped/ship-like product evidence여야 한다.
  3. G4 alumni 인터뷰에는 “AI product 포트폴리오에서 control/override, accessibility, engineering adoption까지 평가받았는가?”를 추가한다.

4. 사용자 포트폴리오로 번역할 체크리스트

첫 공개 case study는 아래 10개 질문에 답해야 한다.

  1. 어떤 사용자/비즈니스 문제였나?
  2. 문제를 어떻게 발견했나? 리서치/데이터/현장 관찰 근거는?
  3. 내가 직접 소유한 범위는 어디까지인가?
  4. 초기 flow는 왜 부족했나?
  5. 어떤 대안을 비교했고, 왜 버렸나?
  6. 최종 flow/UI는 어떻게 복잡성을 줄였나?
  7. PM/Engineer/Stakeholder와 어떤 trade-off를 조정했나?
  8. 결과를 어떤 지표나 정성 신호로 확인했나?
  9. design system/component 관점에서 재사용 가능한 것은 무엇인가?
  10. AI/개발/프로토타이핑을 어디에 사용했고, 인간 판단은 어디에 들어갔나?
