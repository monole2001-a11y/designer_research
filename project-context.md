## 1. 프로젝트 개요

### 무엇인가

미국 프로덕트 디자이너 채용 시장의 2026년 현황과 전망을 종합한 단일 HTML 리서치 리포트.  
취업 준비 디자이너가 "지금 이 직무의 현실이 무엇인가"를 한 페이지에서 파악할 수 있도록 설계됐다.

### 왜 만들었나

기존에 흩어진 자료들 — Layoffs.fyi의 레이오프 수치, Figma Survey의 AI 트렌드, 대학 사이트의 학비, LinkedIn JD 분석 — 을 하나의 일관된 시각 언어로 통합하기 위해 제작됐다. 특히 주니어 취준생이 "어디서 시작해야 하는가"에 답하는 실용적 가이드 역할을 목표로 한다.

### 독자층

| 독자 | 주요 관심 섹션 |
|------|--------------|
| 주니어 취준생 (0–2년) | 섹션 5 (역량), 섹션 9 (학교), 섹션 11 (전략) |
| 미드레벨 전환 고려자 | 섹션 5, 섹션 6 (난이도), 섹션 4 (연봉) |
| 시니어 / Staff | 섹션 3 (AI 영향), 섹션 8 (레이오프) |
| 지역 선택 고민 중 | 섹션 5 (도시별 현황), 섹션 4 |
| 석사 진학 고민 중 | 섹션 10 (학교 가이드) |

## 2. 파일 구조

```
with_claude/
├── product-designer-outlook-2026.html   # 최종 리포트 (단일 파일)
├── design-guide.md                      # CSS·레이아웃·컴포넌트 구현 사양
```

### 세 가지 가이드 파일의 역할 분담

| 파일 | 질문 | 대상 |
|------|------|------|
| `design-guide.md` | "이 컴포넌트를 어떻게 HTML/CSS로 만드나?" | HTML 편집자, 디자이너 |
| `content-guide.md` | "이 수치는 어디서 왔고, 언제 바꿔야 하나?" | 데이터 업데이터, 리서처 |
| `project-context.md` | "왜 이렇게 만들었고, 어떻게 발전해왔나?" | 신규 기여자, 프로젝트 인수인계 |

## 4. 리서치 방법론

### 데이터 수집 원칙

| 우선순위 | 출처 유형 | 예시 |
|---------|---------|------|
| 1차 | 기업 공식 발표, 정부 통계 | BLS 직업 전망, Meta/Microsoft 공식 보도자료 |
| 2차 | 집계 서비스 | Layoffs.fyi, TrueUp, Glassdoor, levels.fyi |
| 3차 | 신뢰도 높은 언론 | TechCrunch, CNBC, Computerworld |
| 4차 | 커뮤니티·전문가 뉴스레터 | Lenny's Newsletter, Her UX Path, UXfolio |

### 불확실한 수치 표기 규칙

- 정확한 수치가 확인된 경우 → 정수 표기 (예: `$110,000`)
- 소스 간 범위가 다를 경우 → 범위 표기 (예: `$65K–$85K`)
- 추정치·근사치인 경우 → `~` 앞붙임 (예: `~$102K`, `~16,000명`)
- 확인 불가능한 경우 → 해당 셀 비워두거나 `—`

### 지역 데이터 처리 방식

도시별 연봉은 출처마다 기준이 다르다 (base only vs. total comp, 개인 보고 vs. 기업 공시).  
섹션 5 (도시별 현황)에서는 다음 우선순위를 적용했다:

1. levels.fyi — total comp 기준 (빅테크 포함 시 높게 나옴)
2. Built In / Glassdoor — 중간 수준
3. ZipRecruiter / Salary.com — base-only, 보수적

범위(`$XX–$XX`)로 표기한 경우 최소값 = Salary.com 기준, 최대값 = levels.fyi 기준이다.

## 5. 설계 의사결정 기록

### 단일 HTML 파일로 구성한 이유

- 배포·공유·인쇄가 파일 하나로 완결
- 외부 의존성(CDN, 서버) 없이 오프라인 열람 가능
- GitHub Gist 또는 이메일 첨부로 즉시 배포 가능

### 한국어로 작성한 이유

- 주요 독자가 한국어 사용자 (한국계 미국 취준생 포함)
- 테이블 헤더·기업명·학교명은 영어 원어 유지 (검색 편의)

### 미국 중심 데이터로 전환한 배경

초기 버전에는 한국 대학 학과 정보가 포함되어 있었으나, 미국 취업 시장 데이터와의 연계성이 낮고 업데이트 부담이 높아 삭제했다.  
한국 연봉 데이터(섹션 4)는 비교 참조용으로 유지한다.

### 섹션 5 (도시별 현황)를 추가한 배경

"같은 Senior라도 SF와 Austin에서 연봉이 얼마나 다른가"라는 질문이 취업 결정에 직접 영향을 미치기 때문에 추가했다.  
단순 연봉 비교를 넘어 구매력 등가(Purchasing Power Parity) 기준으로 비용 대비 실질 가치를 보여주는 것이 이 섹션의 핵심이다.

### AI 대체 프레이밍에 대한 편집 방침

섹션 3의 핵심 메시지 — **"AI는 디자이너를 대체하는 것이 아니라 '분류'하고 있다"** — 는 의도적으로 과장·공포 서사를 피하고 데이터 기반 현실을 제시하는 방향으로 설정됐다.  
이 논조는 보고서 전체에서 일관되게 유지되어야 하며, 업데이트 시에도 변경하지 않는다.

## 9. 기술적 주의사항

### HTML 편집 시

- 섹션 번호(`.num` div 내 숫자)를 바꿀 때 `sed` 단순 치환은 파일 전체에 영향을 미칠 수 있다.  
  반드시 `grep -n`으로 정확한 라인 번호를 먼저 확인한 후 라인 지정 방식으로 수정:
  ```bash
  grep -n '<div class="num">5</div>' product-designer-outlook-2026.html
  # 결과 라인번호를 확인 후:
  sed -i '' '776s/<div class="num">5<\/div>/<div class="num">6<\/div>/' product-designer-outlook-2026.html
  ```

- 섹션 삽입 시 번호 순서: **높은 번호부터** (10→11, 9→10, ..., 5→6) 순서로 바꿔야 이중 치환이 발생하지 않는다.

### CSS 수정 시

- 색상값은 `:root` 블록의 CSS 변수만 수정한다. 인라인 hex 직접 수정 금지.
- 새 컴포넌트 추가 시 기존 `.stat-card`, `.insight-card` 패턴을 먼저 확인하고 재사용 가능하면 신규 CSS 추가 없이 사용한다.

### 데이터 업데이트 시

- 수치가 보고서 여러 곳에 중복되는지 **섹션 6의 일관성 체크포인트 표**를 반드시 참조한다.
- 기준일을 변경할 경우 Hero 영역의 메타 텍스트와 섹션 9 (레이오프) 상단 주석을 동시에 수정한다.
